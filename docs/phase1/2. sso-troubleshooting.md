
# SSO Troubleshooting Guide

This document covers how to diagnose and resolve common SSO failures using Okta's System Log, with real examples from this lab environment.

---

## The Okta System Log

The System Log is the first place to look when an SSO flow fails. It captures every authentication event with full context — actor, target app, IP address, error code, and failure reason.

**Location:** Okta Admin Console → **Reports → System Log**

Key event types to filter for:

| Event Type | Meaning |
|---|---|
| `user.authentication.sso` | SSO assertion generated/attempted |
| `user.session.start` | User session created (post-authentication) |
| `user.authentication.auth_via_mfa` | MFA step completed or failed |
| `policy.evaluate_sign_on` | Authentication policy evaluated |

## Common Failure Scenarios

### 1. Redirect URI Mismatch (OIDC)

**Symptom:** Okta returns a `400 Bad Request` — "The redirect_uri parameter must be a Login redirect URI in the client app settings"

**Cause:** The redirect URI sent in the `/authorize` request does not exactly match any URI registered in the Okta app configuration.

**Example encountered in this lab:** The Okta app was initially configured with `http://localhost:3000/callback` but the Express application uses `http://localhost:3000/authorization-code/callback`. The trailing path difference caused the 400 error.

**Resolution:** Update the Sign-in Redirect URI in the Okta app's General tab to match exactly what the application sends.

**How to diagnose in System Log:** Filter for `user.authentication.sso` with outcome `FAILURE` → expand the event → the `debugContext.debugData.requestedRedirectUri` field shows what the app sent vs what was registered.

### 2. User Not Assigned to Application

**Symptom:** Okta returns a `400 Bad Request` — "User is not assigned to the client application"

**Cause:** The authenticated Okta user does not have an active assignment to the target application.

**Resolution:** Okta Admin → Applications → [App Name] → Assignments → Assign to People or Group.

**System Log indicator:** `access_denied` error code on the `user.authentication.sso` event.

---

### 3. SAML Username Mismatch

**Symptom:** SSO redirects complete but the SP returns an error (e.g. Salesforce "Insufficient Privileges" or "Invalid User")

**Cause:** The username asserted in the SAML NameID does not match any user in the SP's directory, or matches a user with insufficient permissions.

**Example encountered in this lab:** Salesforce Developer Edition auto-generates usernames in the format `firstname.lastname.hash@agentforce.com`. The Okta app assignment must be configured with this exact username, not the user's Okta login email.

**How to diagnose:**
1. Check Okta System Log — if the event shows `SUCCESS`, the assertion was sent correctly and the issue is SP-side
2. Check the SP's own login history — in Salesforce: Setup → Login History
3. Compare the NameID value in the assertion against the SP user record

**Key principle:** A successful Okta System Log event means Okta did its job. Any subsequent error is at the SP layer and must be diagnosed using the SP's own logs.

**What to look for in a failed event:**
- **Outcome: FAILURE** — confirms the failure was at the Okta authentication layer
- **Reason** — e.g. "Invalid credentials", "Account locked", "MFA required"
- **Actor** — confirms which user attempted authentication
- **Client IP** — useful for identifying suspicious login attempts from unexpected locations

## Distinguishing IdP vs SP Failures

This is a common interview question and a genuine day-one diagnostic skill:

| Scenario | Okta System Log | SP Login Log |
|---|---|---|
| Wrong password | `FAILURE` — Invalid credentials | No entry (request never reached SP) |
| User not assigned | `FAILURE` — access_denied | No entry |
| Redirect URI mismatch | `FAILURE` — 400 error | No entry |
| Successful assertion, SP error | `SUCCESS` | `FAILURE` — SP-specific reason |
| Successful assertion, SP accepts | `SUCCESS` | `SUCCESS` |

The pattern is clear: if Okta shows `SUCCESS` and there's still an error, the investigation moves to the SP's logs. If Okta shows `FAILURE`, the fix is at the IdP layer.

## Summary of Issues Encountered and Resolved in This Lab

| Issue | Layer | Resolution |
|---|---|---|
| Redirect URI mismatch (OIDC) | Okta (IdP) | Updated redirect URI in Okta app config to match app |
| User not assigned | Okta (IdP) | Assigned user to app in Okta Assignments tab |
| SAML username mismatch | SP mapping | Updated Okta assignment with exact Salesforce username |
| Post-SSO landing page restriction | SP (OrgFarm org) | Confirmed auth success via System Log and Login History; documented as org-level restriction |
