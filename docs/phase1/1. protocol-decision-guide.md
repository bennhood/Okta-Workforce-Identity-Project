# Protocol Decision Guide: SAML 2.0 vs OIDC

When configuring SSO in an enterprise environment, the choice between SAML 2.0 and OIDC is one of the first decisions an IAM engineer makes. This document outlines the key differences and when each protocol is appropriate.

---

## SAML 2.0

**SAML (Security Assertion Markup Language)** is an XML-based protocol developed in the early 2000s, designed primarily for enterprise browser-based SSO. It remains the dominant protocol for integrating legacy enterprise SaaS applications.

**How it works:**
1. A user attempts to access the Service Provider (SP) - e.g. Salesforce
2. The SP redirects the user to the Identity Provider (IdP) - e.g. Okta - with an `AuthnRequest`
3. The IdP authenticates the user and returns a signed XML **SAML Assertion** to the SP via the browser
4. The SP validates the assertion, extracts the user attributes, and grants access

**When to use SAML:**
- Integrating with enterprise SaaS applications that predate modern OAuth standards (Salesforce, ServiceNow, Workday, SAP)
- Environments where the SP does not support OAuth 2.0 or OIDC
- Federating with external partner organisations using their own IdP

---

## OpenID Connect (OIDC)

**OIDC** is an identity layer built on top of OAuth 2.0, published in 2014. It uses JSON-based tokens (JWTs) and is the standard for modern cloud-native applications and APIs.

**How it works (Authorization Code Flow):**
1. The user clicks login in the application (Relying Party)
2. The app redirects the user to the IdP's `/authorize` endpoint with a `code_challenge`
3. The IdP authenticates the user and returns an **authorization code** to the app's callback URL
4. The app exchanges the code for an **ID token** and **access token** server-side
5. The ID token contains the user's claims (identity); the access token is used for API calls

**When to use OIDC:**
- Modern web applications, mobile apps, and APIs
- Scenarios requiring API access in addition to authentication
- Consumer-facing identity or B2C use cases
- Any application built after approximately 2015 with native OAuth 2.0 support

---

## Side-by-Side Comparison

| Factor | SAML 2.0 | OIDC |
|---|---|---|
| Token format | XML (SAML Assertion) | JSON (JWT) |
| Transport | HTTP POST / Redirect binding | HTTP redirects + back-channel token exchange |
| Primary use case | Enterprise browser SSO | Modern apps and APIs |
| Mobile support | Poor | Native |
| API authorisation | No | Yes (via access token) |
| Complexity | Higher | Lower |
| Typical apps | Salesforce, ServiceNow, Workday | Slack, GitHub, custom apps |
